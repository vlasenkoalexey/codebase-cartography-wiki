relevant entry here, then drill into its page. See [`log.md`](log.md) for the chronological history.

Its signature feature is **heuristic dynamic-dispatch edge synthesis** — ~30 framework-specific passes (Redux thunks, Laravel events, Go implicit interfaces, C++ vtables, React re-render, RN↔native bridges) that recover the `provenance:'heuristic'` edges a static parse structurally misses — layered over a confidence-tiered name-based resolver. | **tree-sitter-analyzer** | [overview](code/tree-sitter-analyzer/overview.md) | `c5a4ae0` | Code intelligence grounded entirely on **tree-sitter parse trees** — no SCIP, no graph DB: a uniform data model shared by 21 language plugins, and a **family-gated** call graph whose only cross-language defense is a small `languages_compatible` check applied exactly where an unscoped project-wide name search needs it (symmetric for JS/TS dialects, directional for C/C++/Objective-C) — plus an independent second application of the same gate at the PR-review impact layer. Persistence is two flat SQLite tables (`ASTCache` + `EdgeStore`), no graph DB. An MCP/CLI surface (`FacadeTool` consolidating 55+ inner tools into 8) self-reports confidence via verdict envelopes and UML-diagram trust labels. Its README benchmarks the family-gating payoff head-to-head against "CodeGraph" (the codegraphcontext/FalkorDB lineage already in this wiki): ~390× fewer cross-language call-graph mis-wires while resolving 3× more call edges. 24 concept pages + 4 doc-concepts over 43,362 symbols (100% represented). |
| **gitgalaxy** | [overview](code/gitgalaxy/overview.md) | `845400b` | The survey's **no grounding substrate** control group — a whole-repo static-analysis engine that is explicitly **AST-free and LLM-free** (its own README: the "blAST" engine, "Bypassing LLMs and ASTs"): a hand-authored, per-language regex "grammar" (`LANGUAGE_DEFINITIONS`, 57 languages) drives a five-mode heuristic function-slicer (`StructuralExtractor`) behind an optical-pipeline metaphor (Aperture Filter → Prism → Detector), calibrated by a project-manifest scanner (GuideStar) and validated by adversarial ReDoS/hallucination test gauntlets instead of a compiler or a citation linter. Structural signal is *counted*, not *resolved* — no cross-file symbol table, no call graph, no dataflow — and hand-tuned formulas turn those counts into ~300 tunable "risk exposure" metrics, with exactly one bolted-on XGBoost classifier (`SecurityAuditor`) crossing the no-model line. Verification found real bugs along the way: a dead "incremental scan" code path, a dead secrets-bypass branch, and a doc/code mismatch. 16 concept pages + 26 doc-concepts over 1,189 symbols (100% represented). |

## Cross-repo concepts
Host vocabulary (`wiki/concepts/<key>.md`), wired by the `wikify-connect-repo` skill: each concept page
links **down** to every repo's implementation, and each implementation links back up. This is where the
survey's comparison lives — grounded in real symbols, not hand-waving.

| Concept | Page | Implemented in | The comparison |
|---|---|---|---|
| **SCIP grounding** | [scip-grounding](concepts/scip-grounding.md) | wikify-repo (2), codegraphcontext (2) · graphify: **none** | A **spectrum**, not a binary. wikify-repo grounds on real SCIP *always* (no SCIP symbol → no citation). **codegraphcontext** ships a fully-wired SCIP path (`scip_pb2` + `_build_graph_from_scip`) but it is **opt-in** — gated on `SCIP_INDEXER=true`, off by default — a precision layer over its tree-sitter default, not the substrate. graphify's `scip_ingest` is a dormant, un-wired skeleton — it grounds on AST instead. |
| **Symbol/call graph** | [symbol-graph](concepts/symbol-graph.md) | wikify-repo (2), graphify (2), understand-anything (7), codegraphcontext (10) | The graph as an *intermediate* that drives LLM synthesis (wikify-repo) vs. the graph *as the product* users query — graphify (query/path/explain), understand-anything (a KnowledgeGraph behind a dashboard), and **codegraphcontext**, the purest case: a property graph an agent queries directly over MCP, built **compiler-lessly** (tree-sitter + heuristic call resolution with confidence-tiered edges), no SCIP required. |
| **Multi-language extraction** | [multi-language-extraction](concepts/multi-language-extraction.md) | wikify-repo (2), graphify (3), understand-anything, codegraphcontext, claude-context (2) | SCIP-first with an AST *fallback floor* (wikify-repo) vs. AST-only, `detect`-routed breadth (graphify — no SCIP) vs. per-language extractors (understand-anything, codegraphcontext). claude-context is the odd one out: it tree-sitter-parses 9 grammars **only to place chunk boundaries** for embedding (with a LangChain char-splitter fallback floor) — extraction to bound the *retrieval unit*, not to build a symbol table. |
| **Incremental reconcile** | [incremental-reconcile](concepts/incremental-reconcile.md) | wikify-repo (2), graphify (3), understand-anything (3), codegraphcontext (2), openwiki (1), claude-context (4) | Reconcile against a *commit pin* (wikify-repo) vs. *continuous* live-working-tree reconcile (graphify) vs. *structural-fingerprint* rebuilds (understand-anything). openwiki is the **prose-shaped** outlier (diff `gitHead..HEAD` → LLM surgical edit → content-hash baseline, no symbol/graph diff). claude-context reconciles the **vector index**: a shallow **Merkle-DAG of content hashes** is the tripwire, and only the changed delta is re-chunked and re-embedded — driven continuously by the MCP server's 5-min sync loop. |
