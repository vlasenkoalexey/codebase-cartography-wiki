# Multi-language extraction

**The survey axis.** Real repos are polyglot, so a comprehension tool must extract symbols and
relationships from many languages — and degrade gracefully when a language has no first-class
support. This page compares each tool's *extraction strategy and its fallback floor*.

## How the implementations differ
- **wikify-repo — SCIP-first, AST-fallback.** A `languages` registry maps each language to its SCIP
  indexer; that indexer is the preferred path (precise, compiler-grade). When no indexer is
  available or it fails, an `ast_fallback` provides a symbol-recovery *floor* so the language is
  still represented rather than dropped. Precision first, coverage guaranteed by the fallback.
- **graphify — AST-first, detect-routed, no SCIP.** `detect` classifies each file (code vs. prose vs.
  media, language by suffix/shebang/manifest), then `extract` dispatches to per-language AST extractors
  built on a shared `extractors-base` (stable node identity, id canonicalization). The long tail of
  languages is covered entirely by hand-written AST walkers, and non-code inputs route to an LLM
  extractor. graphify does **not** use SCIP in practice — its `scip_ingest` module is a dormant,
  un-wired skeleton (see [scip-grounding](scip-grounding.md)).

> [!inferred]
> The strategies diverge at the grounding floor: wikify-repo prefers real SCIP and treats AST as the
> safety net, whereas graphify has no SCIP path at all and relies on deterministic AST extraction as
> its primary breadth mechanism. graphify additionally owns a *routing* step (`detect`) because it
> ingests non-code sources too, which wikify-repo (code-only) does not need.
