# Incremental reconcile

**The survey axis.** A comprehension artifact is only useful if it stays current, and full rebuilds
are too expensive to run on every change. These tools each rebuild only the delta — but they reconcile
against different notions of "changed". This page compares *what triggers a rebuild* in each.

## How the implementations differ
- **wikify-repo — reconcile against a commit pin (build-time, ref-driven).** Its `state` records
  what was built at which commit; `diff` computes which symbols moved between the old pin and a new
  `--ref`, so only the affected concept pages rebuild. The operation is idempotent: re-running
  converges, and a version bump rebuilds just the changed symbols. Reconciliation is a deliberate,
  invoked build step tied to an explicit commit.
- **graphify — reconcile against a live working tree (continuous, event-driven).** A
  content-addressed `cache` (hash of file contents + path) skips re-extracting unchanged files; a
  `watch` daemon debounces filesystem events and re-ingests touched files; and git `hooks`
  re-ingest on commit/checkout. Reconciliation is continuous and automatic — the graph tracks the
  working tree as you edit.
- **openwiki — reconcile against a git-commit diff, prose-shaped (the outlier).** OpenWiki has no
  symbol graph or fingerprint index to diff; its artifact is human-readable Markdown. On `update` it
  records the last successful run's `gitHead` in `openwiki/.last-update.json`, computes
  `git log <gitHead>..HEAD --name-status`, and hands that changed-files list to an LLM agent that is
  *prompted* to edit only the pages those changes touch. The reconcile boundary is therefore advisory
  (prompt-enforced, not mechanically gated), and "did anything change" is decided by a whole-tree
  **content hash** of `openwiki/` — the baseline advances only when the docs actually changed. See
  [openwiki · Agent runtime](../code/openwiki/concepts/openwiki-agent-index.ts.md).

> [!inferred]
> Same goal (rebuild only the delta), opposite cadence: wikify-repo reconciles *point-in-time* at an
> explicit commit (reproducible, pinned), while graphify reconciles *continuously* against whatever
> the working tree currently is (always-on, live). The trade is reproducibility/pinning vs.
> immediacy.
